# Cross-package accessibility — `global` discipline
Even within the same `plse` namespace, 2GP managed packages enforce visibility across package boundaries based on the **declared visibility modifier on the metadata**, not just the namespace wall. A `public` class in Package A is invisible to Apex code compiled into Package B even if both are in `plse`. This is by design — it lets you version packages independently without accidentally exposing internals to every downstream extension.

Therefore the following types **must be declared `global`** in Core so other Pulse packages can reference them:

- `PulseTool` (interface) — implemented by Pulse Tools – Core tools.
- `PulseToolModels` and all inner classes (`Context`, `Result`) — tool impls construct/return these.
- `PulseAiProvider` (interface) — implemented by Pulse AI – Anthropic.
- `PulseAiModels` and all inner classes (`Request`, `Response`, `Message`, `ToolCall`, `ToolDefinition`, `Usage`).
- `PulseConversationAdapter` (interface) — implemented by Pulse Conversations – Recall.
- Any input/output DTOs on `PulseConversationAdapter` — make globally-accessible.
- `SchemaDeploymentGateway` (abstract class) and its inner DTOs (`FieldSpec`, `DeploymentPlan`, `DeploymentResult`) — if we expect third-party deployer backends in a later wave, it should already be global.

Update the `public class`/`public interface` declarations to `global class`/`global interface`. Method signatures exposed across packages must also be `global`. Inner class members referenced across package boundary must be `global`. Constructors that cross-package callers use must be `global`. Tests stay `@isTest private`.

**Do not make everything global** — only the surface that extensions legitimately need. Runtime internals like `WorkflowRuntime`, `HitlService`, `PulseAgentRunner`, `ProjectionWriteService`, `SchemaProvisioningService`, `ToolingApiDeploymentGateway` stay `public` (they are Core-internal; extensions do not construct or subclass them).
