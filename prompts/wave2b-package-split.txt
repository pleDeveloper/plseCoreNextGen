# Cross-package accessibility requirements
Extension packages implement interfaces defined in Core. In Salesforce Apex, a `public` class is only visible inside its own namespace. Since Core is namespaced (`plse`) and extensions are not, the following types **must be declared `global`** in Core so extensions can reference them:

- `PulseTool` (interface) — because `PulseEmailTool`, `PulseRecordUpdateTool`, `PulseExternalApiTool` implement it from the Tools-Core package.
- `PulseToolModels` and all of its inner classes (`Context`, `Result`) — tool impls construct these.
- `PulseAiProvider` (interface) — because `AnthropicAdapter` implements it from the AI-Anthropic package.
- `PulseAiModels` and all of its inner classes (`Request`, `Response`, `Message`, `ToolCall`, `ToolDefinition`, `Usage`).
- `PulseConversationAdapter` (interface) — because `RecallAdapter` implements it from the Conversations-Recall package.
- Any input/output DTOs on `PulseConversationAdapter` — make globally-accessible DTOs.
- `SchemaDeploymentGateway` (abstract class) and its inner DTOs (`FieldSpec`, `DeploymentPlan`, `DeploymentResult`) if we expect third-party deployer backends.

Update the `public class`/`public interface` declarations to `global class`/`global interface`. Method signatures exposed to extensions must also be `global`. Inner class members referenced across namespace must be `global`. Tests in Core stay `@isTest private` — tests do not need `global`.

Do not make everything global — only the surface that extensions legitimately need. Runtime internals like `WorkflowRuntime`, `HitlService`, `PulseAgentRunner`, `ProjectionWriteService`, `SchemaProvisioningService` stay `public` (they are consumed within Core only).
