# Problem statement
`sf project deploy start -o pulse-core-next-dev --dry-run` currently returns Failed. Six unique errors live in your lane:

- `SchemaProvisioningService` — "Invalid type: Metadata.CustomField", "Invalid type: Pulse_Deployment_Status__e", "Method does not exist or incorrect signature: addMetadata(Metadata.CustomField)", "Method does not exist or incorrect signature: publish(Pulse_Deployment_Status__e)", "Variable does not exist: Metadata.FieldType", "Variable does not exist: cf"
- `SchemaProvisioningServiceTest` — "Dependent class is invalid and needs recompilation"

## Likely root cause
The platform event `Pulse_Deployment_Status__e` is currently located at:

`force-app/main/default/platformEvents/Pulse_Deployment_Status__e.object-meta.xml`

This is NOT a recognized DX source directory for API 66.0. Platform events must live under `objects/<EventName>/` just like other SObjects — and the fields that the Apex code references (`Deployment_Request_Id__c`, `Status__c`, `Message__c`, `Target_Object__c`) must live as separate files in a sibling `fields/` subdirectory.

When `sf` can't locate the platform event, the Apex compiler can't resolve `Pulse_Deployment_Status__e` as a valid SObject type. The Apex compiler typically bails on the first unresolved type and reports every subsequent `Metadata.*` reference in the same file as cascading "Invalid type" errors even though the `Metadata` namespace itself is fine in scratch orgs. Fixing the platform event layout is likely sufficient to clear all six errors — but verify with a dry-run after you move it.
