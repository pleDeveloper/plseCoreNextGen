# Goal — build a real Tooling API REST gateway

The gateway will use the Tooling API REST endpoint to create `CustomField` records via HTTP callout through a Named Credential that points back at the same org (self-org callout pattern).

## API contract
Endpoint:  `POST /services/data/v62.0/tooling/sobjects/CustomField`

Named Credential: `callout:Pulse_Self_Org_Tooling` (configured manually per-org; do NOT try to deploy it — see the docs file below).

Request body for a Text field (example):
```
{
  "FullName": "Account.Authorization_Number__c",
  "Metadata": {
    "type": "Text",
    "label": "Authorization Number",
    "length": 80,
    "required": false
  }
}
```

Supported types map (fieldType → Tooling API `Metadata.type`):
- `"Text"` → `"Text"` with `length` (default 255)
- `"LongTextArea"` → `"LongTextArea"` with `length` (default 32768) and `visibleLines` (default 6)
- `"Number"` → `"Number"` with `precision` (default 18) and `scale` (default 0)
- `"Currency"` → `"Currency"` with `precision`/`scale` (defaults 18/2)
- `"Percent"` → `"Percent"` with `precision`/`scale` (defaults 5/2)
- `"Date"` → `"Date"`
- `"DateTime"` → `"DateTime"`
- `"Checkbox"` → `"Checkbox"` with `defaultValue: false`
- `"Picklist"` → Unsupported for Wave 2. Throw an `UnsupportedOperationException` with message `'Picklist CustomField creation via Tooling API requires a valueSet payload — deferred to a later wave.'` so the admin hears a honest error instead of silently getting the wrong field type.

Success response (HTTP 201):
```
{ "id": "00N...", "success": true, "errors": [] }
```

Failure response (HTTP 400):
```
[{ "message": "...", "errorCode": "DUPLICATE_VALUE" }]
```
